# Dev Journal — 开发日志生成器

写完代码之后，把"这次到底改了什么、为什么这么改、对协作方有什么影响、还有哪些坑"沉淀成一份给后人看的 markdown 文档。

## 何时启用

满足以下任一情况就启用本 skill：

- 用户显式要求："写 journal"/"生成开发日志"/"journal 这次改动"/"总结这次修改"/"记录本次变更"
- 用户准备合并 / 提交 / 推送代码前的"收尾"环节
- 用户完成一段较大重构 / 新增功能 / 修复关键 bug 后的复盘

## 工作流程

按顺序执行 4 步，每步完成后再进入下一步。

```
- [ ] Step 1: 收集信息源
- [ ] Step 2: 与用户确认不确定项
- [ ] Step 3: 按模板生成 journal 内容
- [ ] Step 4: 写入 <project_root>/.journal/{YYYY-MM-DD}-{summary}.md
```

### Step 1：收集信息源

按优先级合并以下 4 个来源：

1. **工作区改动**（默认必查）

   ```bash
   git status --porcelain
   git diff --stat
   git diff
   ```

2. **当前对话**：直接利用本会话中讨论过的设计动机、权衡、bug 复现、决策点。无需重新询问已经明确过的内容。

3. **与主分支对比的独有提交**（自动尝试）

   ```bash
   # 检测 base：依次尝试 main / master / develop / trunk
   git rev-parse --verify --quiet main || git rev-parse --verify --quiet master || ...
   git log <base>..HEAD --oneline --no-merges
   git diff <base>...HEAD --stat
   ```

4. **用户显式提供的 commits**：如果用户给出了 hash 列表，逐个 `git show --stat <hash>` 和 `git show <hash>` 提取。

> **当用户提供的 commits 与工作区有重叠时**：以用户显式提供的为准；同时把"工作区未提交改动"作为补充说明。

### Step 2：与用户确认不确定项（Fallback 询问规则）

只在以下情况主动询问，能自行判断的不要打扰：

| 情况 | 必须问 |
|------|--------|
| 检测不到 base 分支（无 main/master/develop/trunk） | 问用户主分支名 |
| 工作区干净 + 当前分支等于 base + 没有用户提供 commits | 问用户提供 commit hashes 或文件范围 |
| 无 API/接口文件改动，但用户提到了协作方 | 问是否需要填"给协作方"章节 |
| summary 短语在 2-3 个候选间无法决断 | 给候选让用户选 |
| 涉及破坏性变更（删 API、改字段类型），但无法判断是否对外 | 必须问 |

询问应一次性合并多个问题，避免来回打断。

### Step 3：生成 journal 内容

严格按四段式模板生成（完整模板见下文 §模板，也可读取 `templates/journal-template.md`）。

四段都是**必有结构**，但每段内部"无内容则写空"或"标注 N/A"：

1. **本次修改总结**
2. **给后人快速理解**
3. **给协作方**（接口/数据契约/调用示例/预期数据）
4. **Review**（潜在问题与优化指南）

> 第三段（协作方）在纯内部重构、无接口变更时，可以只保留一行："N/A — 本次为内部重构，无对外接口变更。" 但不要删掉这一段，保持结构稳定，方便检索。

### Step 4：写入文件

**目录**：`<project_root>/.journal/`（不存在则创建；同时在 `.gitignore` 中检查是否已忽略——如未忽略，**询问**用户是否要 git 跟踪此目录，不擅自加 `.gitignore`）。

**文件名规则**：

- 日期：`YYYY-MM-DD`（按本地时区）
- summary：中文短语，10–20 字符；剔除文件系统不允许的字符 `/ \ : * ? " < > |`；空格替换为 `-`
- 拼接：`{date}-{summary}.md`，例如 `2026-05-12-修复登录态过期边界.md`
- **冲突处理**：同名已存在 → 追加 `-HHmm`，例如 `2026-05-12-1456-修复登录态过期边界.md`；仍冲突则再追加秒 `-HHmmss`

写入完成后，向用户输出：

```
✓ Journal written to .journal/2026-05-12-修复登录态过期边界.md
  - 修改文件：12
  - 新增 / 修改 / 删除：+ 234 / ~ 56 / - 12
```

## 模板（四段式）

> 同样的模板存了一份在 `templates/journal-template.md`，需要细看时再读。

```markdown
# {summary}

**日期**：{YYYY-MM-DD}
**作者**：{git user.name}
**分支**：{branch} → {base-branch}
**涉及提交**：{commit hashes 列表，或"工作区未提交改动"}
**涉及文件数**：{N}（+{add} / ~{modify} / -{delete}）

---

## 一、本次修改总结

> 一句话答案：**这次改了什么？为什么改？**

- 修改背景 / 触发原因（issue / bug / 需求 / 主动重构）
- 核心修改逻辑（按模块/层级分点列出，每点配文件路径）
- 关键设计选择（如果是多选一，说明为什么选了这个）

## 二、给后人快速理解

> 目标：让后续接手的同学在 **5 分钟** 内 get 到关键点。

- **设计动机与权衡**：为什么这么做，不那么做？
- **关键概念 / 术语**：本次引入或重新定义的术语
- **数据流 / 调用链**：（必要时配 mermaid 图）
- **潜在副作用 / 已知限制**
- **相关上下文**：issue / PR / 设计文档链接

## 三、给协作方

> 涉及接口、数据契约、前后端协议、SDK 行为变化时**必填**；纯内部重构填 "N/A"。

### 受影响的接口

| 接口 | 方法 | 变更类型 | 兼容性 |
|------|------|----------|--------|
| `/api/foo` | GET | 修改 | 向后兼容 |

### 调用示例

**请求**：

\`\`\`http
POST /api/foo HTTP/1.1
Content-Type: application/json

{
  "id": "abc",
  "newField": "value"
}
\`\`\`

**响应（成功）**：

\`\`\`json
{
  "code": 0,
  "data": { "id": "abc", "status": "ok" }
}
\`\`\`

**响应（失败）**：

\`\`\`json
{ "code": 40001, "message": "..." }
\`\`\`

### 字段说明

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|

### 客户端 / 调用方需要做的改动

- [ ] 行动项 1
- [ ] 行动项 2

**上线节奏建议**：（同时上 / 先后端 / 先客户端 / 灰度策略）

## 四、Review（潜在问题与优化指南）

> 给未来的自己留一份"已知坑"。

- **潜在问题**
  - [ ] 并发 / race condition
  - [ ] 性能（大数据量、慢查询）
  - [ ] 边界 / 空值 / 错误处理
  - [ ] 安全（鉴权、注入、越权）
  - [ ] 兼容性（旧版本客户端、旧数据）
- **测试覆盖盲区**
- **短期优化建议**
- **长期演进方向**
- **监控 / 灰度建议**
```

## 示例：一份典型 journal（节选）

```markdown
# 修复登录态过期后接口 401 重试死循环

**日期**：2026-05-12
**作者**：iskl
**分支**：fix/login-loop → main
**涉及提交**：a1b2c3d, e4f5g6h
**涉及文件数**：5（+0 / ~5 / -0）

---

## 一、本次修改总结

- **背景**：用户反馈登录态过期后，部分接口会无限重试导致前端卡死。
- **根因**：`request.ts` 的 401 拦截器在刷新 token 失败后没有清空重试队列，
  导致后续请求继续触发刷新流程。
- **修改**：
  - `src/utils/request.ts`：刷新失败时立刻 reject 等待队列
  - `src/utils/auth.ts`：抽出 `clearTokenAndRedirect` 公共方法

## 二、给后人快速理解

- 这套拦截器的核心是 **"刷新中-排队-完成后统一放行"** 模型
- 关键概念：`pendingQueue`（等待刷新的请求）、`isRefreshing`（互斥锁）
- 数据流：

  \`\`\`
  request → 401 → 入队 pendingQueue → 触发 refresh
                                        ↓ 成功
                                     放行 pendingQueue
                                        ↓ 失败
                                     reject + 跳登录页
  \`\`\`

- 已知限制：刷新接口本身 401 时仍会重试 1 次（这是有意的，处理时钟漂移）

## 三、给协作方

N/A — 本次为前端内部修复，无对外接口变更。

## 四、Review

- **潜在问题**
  - [x] 边界：`pendingQueue` 在页面卸载时未清理（已修复）
  - [ ] 并发：多个 tab 同时刷新仍可能产生竞争（已知，待统一 BroadcastChannel）
- **测试盲区**：缺少 "refresh 接口超时" 的 e2e 用例
- **短期优化**：把 token 刷新逻辑迁移到 SW
```

## 输出语言约定

- 默认中文（包括小节标题、列表项、表格表头）
- 代码块、文件路径、命令保持原样
- 用户/项目主语言为英文时自动切英文（保持模板结构不变）

## 不要做的事

- 不要在写完 journal 后改用户的代码 / 提交 / 推送
- 不要在 `.journal/` 目录之外创建文件
- 不要悄悄修改 `.gitignore`
- 不要把 Review 写成"完美无瑕、上线即可"——必须真实列出至少 1 条潜在问题或测试盲区
- 不要丢失四段式结构（即使某段为空也要保留标题 + "N/A"）

## 与其他工具的协作

- 用户已用 git commit/conventional commit 时，可直接引用 commit message 作为 summary 候选
- 项目有 CHANGELOG 时，journal 仍然要写——journal 关注"为什么 + 给谁看"，CHANGELOG 关注"发了什么"
